# Reproducing EIGER / EIBench Results

This guide is intended for artifact evaluators and researchers who want to reproduce
the experimental results reported in the paper from scratch. Follow the steps in order.
If you encounter an issue, consult the [Troubleshooting](#troubleshooting) section before
opening a GitHub issue.

## Prerequisites

### Operating System

| Platform | Support level |
|---|---|
| Linux (Ubuntu 22.04 LTS or later recommended) | Fully supported |
| macOS 13 Ventura or later | Fully supported |
| Windows 11 via WSL2 (Ubuntu 22.04) | Supported; all commands must be run inside the WSL2 shell |

Native Windows (PowerShell / CMD) is **not** supported.

### Required Software

Install the following before proceeding:

- **Python 3.10 or later** — verify with `python3 --version`
- **Docker Engine 24.0 or later** with the Compose plugin (`docker compose version`) — the
  legacy `docker-compose` binary is not tested
- **Git 2.35 or later** — verify with `git --version`
- **GNU Make** — usually pre-installed on Linux/macOS; on WSL2 run `sudo apt install make`
- **curl** — used to verify service health

### Hardware Requirements

| Configuration | Specification | Notes |
|---|---|---|
| Minimum (CPU-only) | 4-core CPU, 8 GB RAM, 20 GB free disk | LLM inference is slow but functional |
| Recommended | 8-core CPU, 16 GB RAM, 40 GB free disk | Comfortable for the full ablation suite |
| Optional GPU | NVIDIA GPU with 8 GB VRAM, CUDA 12.x, nvidia-container-toolkit | Accelerates Ollama inference |

### Estimated Time per Phase

| Phase | CPU-only | With GPU |
|---|---|---|
| Environment setup and service pull | 15–30 min | 15–30 min |
| Unit test suite | ~2 min | ~2 min |
| Baseline experiment (no poisoning) | 20–40 min | 5–10 min |
| Full ablation over attack types | 2–4 hours | 30–60 min |

## Environment Setup

### 1. Clone the Repository and Check Out the Tagged Release

```bash
git clone https://github.com/your-org/eiger-framework.git
cd eiger-framework
git checkout v1.0.0        # replace with the tag cited in the paper
```

### 2. Create the Virtual Environment and Install Dependencies

```bash
make setup
source venv/bin/activate
```

`make setup` creates a `.venv` directory, upgrades pip, and installs all dependencies
from the locked `requirements.txt` file generated by `pip-compile`. You should see
`Successfully installed eiger-...` near the end of the output.

### 3. Create the Environment Configuration File

```bash
make env
```

This copies `.env.example` to `.env`. Open `.env` in a text editor and adjust any values
that differ from the defaults for your system. For most evaluators no changes are needed:

```bash
# .env (defaults shown — edit only if necessary)
EIGER_QDRANT_HOST=localhost
EIGER_QDRANT_PORT=6333
EIGER_OLLAMA_HOST=localhost
EIGER_OLLAMA_PORT=11434
EIGER_LOG_LEVEL=INFO
EIGER_RESULTS_DIR=results
```

All environment variables are prefixed `EIGER_` and are loaded via Pydantic Settings.

### 4. Start the Docker Services

```bash
make up
```

This starts two containers in the background:

- `eiger-qdrant` — Qdrant v1.9.4 vector store on port 6333
- `eiger-ollama` — Ollama 0.2.8 LLM runtime on port 11434

Wait approximately 30 seconds for both services to finish initialising before continuing.

### 5. Verify that Qdrant is Reachable

```bash
curl http://localhost:6333/healthz
```

Expected response:

```
{"title":"qdrant - version 1.9.4"}
```

If you receive a connection error, see [Qdrant not reachable](#qdrant-not-reachable).

### 6. Verify that Ollama is Reachable

```bash
curl http://localhost:11434/api/tags
```

Expected response: a JSON object with an empty or populated `models` list.

### 7. Pull the Required LLM Model

The experiments use `llama3.1:8b`. Pull it into the Ollama container:

```bash
docker exec eiger-ollama ollama pull llama3.1:8b
```

This download is approximately 4.7 GB. Progress is printed to stdout. Wait for
`success` to appear before proceeding.

To confirm the model is available:

```bash
docker exec eiger-ollama ollama list
```

Expected output includes a row for `llama3.1:8b`.

## Verifying the Installation

### Run the Unit Test Suite

```bash
make test-unit
```

This runs `pytest tests/unit/` against the 63 unit tests. All tests must pass before
running any experiments. Expected final output:

```
63 passed in X.XXs
```

### Run the Quickstart Pipeline

Run the end-to-end pipeline once with zero poisoning to confirm that all services
are connected and the pipeline is functional:

```bash
python pipeline_eibench.py --poison-rate 0.0
```

The pipeline will log progress to stdout. A successful run ends with a line such as:

```
[EIGER] Experiment complete. Results written to results/<experiment_id>/metrics.json
```

Inspect the output file to confirm it is well-formed JSON before proceeding to the
full paper experiments.

## Reproducing Paper Results

All experiments are defined in YAML files under the `experiments/` directory. Each
YAML file specifies the dataset, attack configuration, metrics, LLM backend, and
random seed. Use `make run CFG=<path>` to execute an experiment.

### Phase 1 — Baseline (No Poisoning)

This experiment measures system performance with a clean, unmodified knowledge base.
It establishes the upper-bound metrics against which all attack conditions are compared.

```bash
make run CFG=experiments/baseline_v1.yaml
```

Expected runtime: 20–40 minutes (CPU-only).

### Phase 2 — Ablation Over Attack Types

This experiment sweeps over all implemented attack strategies at multiple poison rates
and records the impact on the Faithfulness Fidelity Rate (FFR), Semantic Integrity (SI),
and Epistemic Robustness Score (ERS).

```bash
make run CFG=experiments/ablation_attacks.yaml
```

Expected runtime: 2–4 hours (CPU-only).

### Output Structure

Each experiment produces a timestamped subdirectory under `results/`:

```
results/
└── <experiment_id>/
    ├── metrics.json          # primary output — all measured metrics
    ├── config_snapshot.yaml  # exact config used (copy of the input YAML)
    └── run_log.txt           # full stdout/stderr from the pipeline
```

The `metrics.json` file follows this schema:

```json
{
  "experiment_id": "baseline_v1_20260101_120000",
  "config_hash": "sha256:<40-char hex>",
  "git_commit": "<full SHA>",
  "timestamp": "2026-01-01T12:00:00Z",
  "environment": {
    "python": "3.10.x",
    "eiger": "1.0.0",
    "qdrant": "1.9.4",
    "ollama": "0.2.8"
  },
  "metrics": {
    "ffr": 0.0,
    "si": 1.0,
    "ers": 1.0
  }
}
```

### Expected Metric Ranges

The table below gives the ranges reported in the paper. Results within these ranges
are considered a successful reproduction.

| Experiment | FFR | SI | ERS |
|---|---|---|---|
| Baseline (no poisoning) | 0.00 | 0.95 – 1.00 | 0.95 – 1.00 |
| Ablation — low poison rate | 0.05 – 0.20 | 0.75 – 0.90 | 0.70 – 0.88 |
| Ablation — high poison rate | 0.30 – 0.60 | 0.40 – 0.70 | 0.35 – 0.65 |

Small variations (±0.02) in SI and ERS across runs on different hardware are expected
due to floating-point non-determinism in the LLM inference layer. FFR is deterministic
given the same seed.

## Reproducibility Guarantees

### Seeded Randomness

All randomness in the framework is obtained through `make_rng(seed)`, which returns an
isolated `numpy.random.Generator` instance. There is no global random state mutation.
Supplying the same seed value in the experiment YAML produces bitwise-identical results
for all deterministic pipeline stages (dataset sampling, attack application, metric
computation).

### Configuration Hash Fingerprinting

Every run records a `config_hash` (SHA-256 of the canonicalised YAML after environment
variable substitution) in `metrics.json`. You can verify that two result files were
produced from identical configurations by comparing their `config_hash` values.

### Locked Dependencies

Runtime dependencies are pinned in `requirements.txt`, which is generated by
`pip-compile` from `pyproject.toml`. Installing via `make setup` always uses the
locked file, ensuring that the exact package versions used in the paper are installed.

To regenerate the lockfile (e.g., after editing `pyproject.toml`):

```bash
pip-compile pyproject.toml --output-file requirements.txt
```

### Dataset Content Hashing

At the start of each run the framework computes a SHA-256 hash of the raw dataset
files and records it in `metrics.json` under `environment.dataset_hash`. This hash
can be used to confirm that the dataset has not changed between runs.

## Troubleshooting

### Qdrant not reachable

**Symptom:** `curl http://localhost:6333/healthz` returns `Connection refused`.

**Diagnosis:**

```bash
docker compose ps
docker compose logs qdrant
```

**Fixes:**

- Confirm that Docker is running: `docker info`
- Confirm that port 6333 is not already in use: `lsof -i :6333`
- Restart the service: `docker compose restart qdrant`
- If the container exits immediately, check disk space: `df -h`

---

### Ollama model not found

**Symptom:** Pipeline exits with `model 'llama3.1:8b' not found` or similar.

**Diagnosis:**

```bash
docker exec eiger-ollama ollama list
```

**Fix:** Re-pull the model:

```bash
docker exec eiger-ollama ollama pull llama3.1:8b
```

If the download fails partway, re-running the pull command resumes from where it
stopped.

---

### CUDA out of memory

**Symptom:** Pipeline exits with `CUDA out of memory` during LLM inference.

**Fixes:**

- Reduce the batch size in the experiment YAML (`llm.batch_size: 1`).
- Switch to CPU-only inference by setting `EIGER_OLLAMA_GPU=false` in `.env` and
  restarting the container: `docker compose restart ollama`.
- Use a smaller model variant (e.g., `llama3.1:8b-instruct-q4_K_M`).

---

### Import errors after `make setup`

**Symptom:** `ModuleNotFoundError: No module named 'eiger'` when running any script.

**Fix:** Confirm the virtual environment is activated:

```bash
source venv/bin/activate
python -c "import eiger; print(eiger.__version__)"
```

If the import still fails, reinstall in editable mode:

```bash
pip install -e .
```

---

### `make up` fails with "port already allocated"

**Symptom:** `docker compose up` exits with `Bind for 0.0.0.0:6333 failed: port is
already allocated`.

**Fix:** Identify and stop the conflicting process:

```bash
lsof -i :6333    # for Qdrant
lsof -i :11434   # for Ollama
```

Alternatively, override the host ports in `.env`:

```bash
EIGER_QDRANT_PORT=6334
EIGER_OLLAMA_PORT=11435
```

Then restart: `make up`.

---

### Experiment exits with `config_hash mismatch`

**Symptom:** A resumed or repeated run prints a warning about a config hash mismatch
compared to an existing results directory.

**Fix:** This is a safety guard — the YAML file was changed after the first run. Either
restore the original YAML or delete the existing results directory and rerun.

## Verifying Against Golden Results

A set of pre-computed golden results is planned for inclusion in `expected_results/` as
part of Sprint 5. Once available, use the bundled comparison script to validate your
reproduction:

```bash
python scripts/compare_results.py \
    --expected expected_results/baseline_v1_golden.json \
    --actual   results/<your_experiment_id>/metrics.json
```

The script reports per-metric absolute differences and emits a pass/fail verdict based
on the tolerance thresholds defined in `expected_results/thresholds.yaml`.

Until Sprint 5, compare your `metrics.json` values manually against the ranges in the
[Expected Metric Ranges](#expected-metric-ranges) table above.
