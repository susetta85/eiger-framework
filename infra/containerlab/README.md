# ContainerLab Infrastructure

This directory contains the ContainerLab topology for the EIBench distributed architecture.

## When to use this

ContainerLab is **not required** for running experiments. It is provided for:
- Network-level threat modeling (if the research extends to network-layer poisoning)
- Multi-node distributed deployments with isolated container networking
- Reproducibility of the original network topology used during research

For development and standard experiment execution, use Docker Compose instead:

```bash
# From the repository root
docker compose up -d
```

## Topology overview

The `topology.clab.yml` defines 9 nodes connected via a virtual Ethernet network:

| Node              | Role                              |
|-------------------|-----------------------------------|
| `controller`      | Orchestrates all pipeline stages  |
| `corpus-builder`  | Layer 1: claim ingestion          |
| `poisoning-engine`| Layer 2: adversarial attacks      |
| `qdrant`          | Layer 3: vector store (port 6333) |
| `retriever`       | Layer 3: dense retrieval          |
| `generator-llama` | Layer 4: Llama 3.1 backend        |
| `generator-mistral`| Layer 4: Mistral backend         |
| `evaluator`       | Layer 5: metric computation       |
| `analytics`       | Layer 6: plotting and reporting   |

## Prerequisites

```bash
# Install ContainerLab
bash -c "$(curl -sL https://get.containerlab.dev)"

# Verify
containerlab version
```

## Deploy

```bash
sudo containerlab deploy -t topology.clab.yml
```

## Destroy

```bash
sudo containerlab destroy -t topology.clab.yml
```

## Notes

- The `clab-eibench-research-net/` runtime directory generated by ContainerLab
  is gitignored and must never be committed (it contains TLS keys and host-specific data).
- Image versions in the topology should be pinned for reproducibility before any
  paper submission (replace `ubuntu:22.04` with a project-specific image).
